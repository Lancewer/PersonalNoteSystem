存储方案详细对比

三个开源项目的实际存储方案对比 + 你的最佳选择

📊 开源项目实际做法

Memos (59.1k⭐)

DB-first + 部分MD导出
  • SQLite/PostgreSQL 主存储
  • 内容直接存DB
  • 附件URL引用

Logseq (42.3k⭐)

纯Markdown文件
  • 本地文件夹直接存储
  • Git版本控制
  • Obsidian原生兼容

Joplin

SQLite + 文件资源
  • DB存笔记元数据
  • .resource/ 存附件
  • 可导出MD

A. 纯 Markdown 文件

最简单，Obsidian直接读取。但搜索和标签过滤慢（10K条笔记≈200-500ms扫描）。

优点

  • 100% Obsidian兼容
  • Git版本控制天然支持
  • 无数据库依赖
  • 备份只需拷文件夹

缺点

  • 大批量查询慢
  • 并发编辑需文件锁
  • 元数据散落于frontmatter
  • RAG需额外扫描索引
混合方案（推荐）
SQLite 主存储
notes (id, content, tags)
resources (id, path, type)
notes_fts (全文搜索索引)
→
异步导出 MD
/export/
├── 2024/01/15/
│ └── memo-001.md
└── attachments/
└── photo.jpg
Obsidian 目录 ← 实时同步 ← 导出层

B. 混合方案（SQLite + MD 导出）

两全其美。DB做快速查询和RAG，MD文件保兼容 Obsidian。Memos 和 Joplin 都采用的思路。

优点

  • 查询快 25x（FTS5索引）
  • 修改方便（精确字段更新）
  • RAG 就绪（存embeddings）
  • Obsidian仍可兼容

缺点

  • 架构略复杂
  • 需维护导出同步逻辑
  • DB文件需定期备份
纯数据库方案
-- 只存DB，不导出MD
CREATE TABLE notes (
  id UUID PRIMARY KEY,
  content TEXT,
  tags JSON,
  created_at TIMESTAMP
);
⚠️ 无法被 Obsidian 直接读取

C. 纯数据库（不推荐）

性能好，但完全锁定。数据被锁在DB里，脱离应用就无法读取。

优点

  • 最佳查询性能
  • 结构化最强

缺点

  • 不兼容任何外部工具
  • 厂商锁定严重
  • 违反私有化理念

🎯 我的推荐：混合方案 B

数据流程：

1️⃣ 输入：PWA 上传记录 → SQLite 入库
2️⃣ 异步：后台任务 → 导出 .md 文件
3️⃣ 附件：存文件系统 → DB 存 path 引用
4️⃣ Obsidian：直接读取 → 导出目录

为什么好？修改笔记时只需更新DB（快），同时后台自动更新MD文件（Obsidian兼容），RAG 时可直接用 DB 中的 embeddings。
